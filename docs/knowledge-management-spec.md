## 一、知识库架构总览

所有知识按以下体系分类管理：

```
workspace/
├── SOUL.md              # 🟢 灵魂文档——身份、核心规则、强制执行机制（最高优先级）
├── USER.md              # 🔵 用户档案——用户信息、偏好、要求（第二优先级）
├── AGENTS.md            # 🟡 工作规范——运行规范、行为准则、记忆管理规则
├── MEMORY.md            # 🟣 长期记忆——经验提炼、决策记录、知识索引
├── TOOLS.md             # ⚪ 工具笔记——环境配置、基础设施、本地设置
├── IDENTITY.md          # ⚪ 身份卡片——名称、头像、基本元信息
├── NOW.md               # ⚪ 当前焦点——正在做的事、待办、最近变更
├── HEARTBEAT.md         # ⚪ 心跳模板——周期性检查任务
│
├── memory/              # 📓 每日日志目录
│   ├── YYYY-MM-DD.md    # 会话历史原始记录
│   └── ...              # 按日期归档
│
├── skills/              # 🛠️ 技能库
│   ├── prompt-optimizer/ # 提示词优化技能
│   ├── VIDEO/            # 视频制作知识
│   ├── short-drama-production/ # 短剧制作
│   └── ...
│
├── docs/                # 📖 规范文档目录
│   ├── knowledge-management-spec.md  # 本文档
│   └── ...
│
├── output/              # 🎬 项目输出（提示词成品等）
├── templates/           # 📋 格式模板
└── references/          # 🔗 参考文件（如有）
```

## 二、知识分类体系

### 2.1 分类维度

所有知识按以下**五个维度**分类存储：

| 维度 | 说明 | 存储位置 |
|------|------|---------|
| **身份** | 我是谁、我的定位、我的核心原则 | SOUL.md |
| **用户** | 用户是谁、用户偏好、用户要求 | USER.md |
| **规范** | 工作流程、行为准则、强制执行机制 | AGENTS.md + SOUL.md（规则部分）|
| **经验** | 学到的教训、决策记录、实战沉淀 | MEMORY.md + memory/ |
| **工具** | API配置、环境变量、设备信息 | TOOLS.md |

### 2.2 分类标签系统

所有记忆条目使用以下标签分类：
- `[身份]` - 身份定位、自我认知
- `[用户]` - 用户信息、偏好
- `[规范]` - 工作流程、行为规则
- `[提示词]` - 提示词相关的规则、技巧、教训
- `[打斗]` - 打斗场景规则
- `[镜头]` - 镜头语言、运镜规则
- `[对话]` - 对话场景规则
- `[审核]` - 审核规避技巧
- `[格式]` - 格式规范、模板
- `[工具]` - 工具配置、环境信息
- `[项目]` - 项目进展、重要决策
- `[教训]` - 犯过的错、改进方案
- `[人物]` - 合作人员信息
- `[配置]` - 系统配置、模型配置

## 三、文档管理规范

### 3.1 文档优先级

| 优先级 | 文档 | 说明 |
|--------|------|------|
| P0 | SOUL.md | 最高，不可违反的核心规则 |
| P1 | USER.md | 用户明确要求 |
| P2 | AGENTS.md | 运行规范 |
| P3 | MEMORY.md | 经验知识 |
| P4 | skills/* | 领域专项知识 |
| P5 | memory/* | 原始日志 |

冲突规则：高优先级 > 低优先级。同优先级下，更新时间最新的为准。

### 3.2 文档更新触发条件

以下情况**必须**更新文档：

| 触发条件 | 更新位置 | 更新内容 |
|----------|---------|---------|
| 用户说"记住这个"、"记一下"等 | MEMORY.md + 对应分类文档 | 提炼核心信息，按分类写入 |
| 用户说"把xxx作为最新版本" | 对应版本文档 | 更新版本号，归档旧版本 |
| 用户提出新规则/新要求 | SOUL.md 或 USER.md | 根据规则类型判断归属 |
| 我犯了错/学到了教训 | MEMORY.md + SOUL.md（永久规则） | 记录错误和改进方案 |
| 完成重要会话 | memory/YYYY-MM-DD.md | 写当日日志，提炼要点 |
| 用户信息变更 | USER.md | 更新用户档案 |
| 环境/配置变更 | TOOLS.md | 更新工具信息 |
| 项目有重大进展 | NOW.md + MEMORY.md | 记录决策和方向 |
| 收到新剧本/剧本修订 | 对应项目文件 | 版本化保存 |

### 3.3 文档内容边界

| 文档 | 应该写什么 | 不应该写什么 |
|------|-----------|-------------|
| SOUL.md | 核心身份、不可违反的铁律、强制执行机制 | 临时信息、一次性事件、详细案例 |
| USER.md | 用户个人档案、明确提出的偏好和要求 | 我的自我认知、规则推导、中间过程 |
| AGENTS.md | 工作流程、运行规范、记忆管理规则 | 视频制作领域知识、镜头规则细节 |
| MEMORY.md | 提炼后的长期经验、重要决策、关键教训 | 原始对话记录、重复内容 |
| memory/YY-MM-DD.md | 会话原始记录、待提炼的素材 | 已经提炼到MEMORY.md的冗余信息 |
| TOOLS.md | API Key、IP地址、设备配置 | 工作流程、行为规范 |

### 3.4 注释规范

- 重大规则更新：在末尾用 `## 🔴 YYYY-MM-DD 标题` 标注
- 临时标记：用 `<!-- TODO: xxx -->` 标记待办
- 版本记录：在文档开头的元信息区记录最后更新时间
- 引用关系：`参考：docs/knowledge-management-spec.md §3.2`

## 四、记忆更新规范

### 4.1 会话日志处理流程

每次对话结束后（或对话自然中断时），按以下流程处理：

```
会话结束
  ├── 扫描本次对话中的重要信息
  ├── 按分类标签归档
  ├── 更新 memory/YYYY-MM-DD.md（原始日志）
  ├── 判断是否需更新 MEMORY.md
  │     ├── 是 → 提炼核心要点，追加到 MEMORY.md
  │     └── 否 → 保留在日志中，后续 review 时再判断
  ├── 判断是否需更新 SOUL.md / USER.md / AGENTS.md
  │     ├── 新规则 → SOUL.md
  │     ├── 用户新偏好 → USER.md
  │     └── 新工作流程 → AGENTS.md
  └── 更新文档索引（MEMORY.md 索引部分）
```

### 4.2 记忆提炼标准

什么信息**值得**提炼到 MEMORY.md：

- ✅ 反复验证过的规则和技巧
- ✅ 导演/用户明确强调的偏好
- ✅ 导致过错误的经验教训
- ✅ 项目中的关键决策
- ✅ 新的格式规范或模板
- ✅ 合作人员的关键反馈

什么信息**不**需要提炼：

- ❌ 一次性对话的客套内容
- ❌ 临时性的中间状态
- ❌ 已经过时的旧规则（需要标记删除）
- ❌ 重复的内容（已有相同记录）

### 4.3 "记住xxx" 处理流程（增强版）

当用户说"记住xxx"、"把xxx作为最新版本"、"记一下"等时：

**Step 1：理解意图**
- 用户想记住什么？规则？偏好？事实？版本？
- 这是新的还是更新旧的？

**Step 2：分类归属**
```
[规则类] → SOUL.md（核心规则）或 AGENTS.md（工作流程）
[用户类] → USER.md（偏好/要求/信息）
[经验类] → MEMORY.md（技巧/教训/决策）
[工具类] → TOOLS.md（配置/环境）
[版本类] → 对应项目文件（版本化存档）
[日志类] → memory/YYYY-MM-DD.md（原始记录）
```

**Step 3：合理性检查**
- 检查是否与现有规则冲突
- 检查是否合理可执行
- 不合理则告知原因并提出替代方案

**Step 4：执行更新**
- 按分类写入对应文档
- 更新 MEMORY.md 中的文档索引
- 记录更新时间

**Step 5：反馈确认**
- 告知用户已更新哪个文档、什么位置
- 说明更新的核心内容

### 4.4 定期维护

**每次心跳（heartbeat）时（最多每天一次）：**
1. 检查是否有未处理的 `memory/` 日志需要提炼
2. 检查 MEMORY.md 是否有过时内容需要标记/清理
3. 检查各文档索引是否完整

**每周（至少一次）：**
1. 通读 MEMORY.md，合并相似条目
2. 清理过时的每日日志
3. 更新 NOW.md 的焦点和待办

## 五、知识索引体系

### 5.1 MEMORY.md 索引结构

MEMORY.md 开头必须包含文档索引（Table of Contents），格式：

```markdown
# MEMORY.md - 长期记忆

## 📋 文档索引

| 分类 | 条目 | 更新时间 | 位置 |
|------|------|---------|------|
| [规范] | 提示词格式规范 | 2026-05-21 | §格式规范 |
| [打斗] | 打斗因果链规则 | 2026-06-08 | §打斗规则 |
| ... | ... | ... | ... |
```

### 5.2 交叉引用

重要信息需要标注交叉引用：
- `→ 详见 SOUL.md §强制执行机制`
- `→ 参考 USER.md §七、语速与时间铁律`
- `→ 参见 skills/prompt-optimizer/SKILL.md`

## 六、版本管理与备份

### 6.1 备份策略

| 备份对象 | 备份时机 | 备份位置 |
|---------|---------|---------|
| SOUL.md | 每次修改前 | /opt/ai_backup/ |
| USER.md | 每次修改前 | /opt/ai_backup/ |
| AGENTS.md | 每次修改前 | /opt/ai_backup/ |
| MEMORY.md | 每周或重大更新前 | /opt/ai_backup/ |
| skills/* | 更新前 | /opt/ai_backup/ |

### 6.2 备份命名规范

```
{文件名}.bak.{YYYYMMDD}_{HHMMSS}
```

示例：`SOUL.md.bak.20260611_152411`
